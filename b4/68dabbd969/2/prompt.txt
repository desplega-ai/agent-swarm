## Analysis: Scheduled Tasks Implementation

### Overview

Scheduled tasks allow recurring agent task creation via cron expressions or fixed intervals. The system stores schedule definitions in a `scheduled_tasks` SQLite table, runs a 10-second polling loop to find due schedules, creates agent tasks from templates, and tracks error state with exponential backoff. Schedules are managed through three MCP tools: create, update, and list.

### 3. Scheduler Engine (`scheduler/scheduler.ts`)

#### Module State (lines 11-12)

Two module-level variables control the polling loop:
- `schedulerInterval` -- the `setInterval` handle
- `isProcessing` -- a mutex flag to prevent concurrent polling cycles

#### `startScheduler()` (lines 182-196)

- Guards against double-start.
- Calls `recoverMissedSchedules()` first, then starts `processSchedules()`.
- Sets up a repeating `setInterval` (default 10,000ms) that calls `processSchedules()`.

#### `processSchedules()` (lines 201-218)

- Checks the `isProcessing` mutex; returns immediately if already running.
- Calls `getDueScheduledTasks()` to get all schedules with `nextRunAt <= now`.
- Iterates and calls `executeSchedule()` for each, catching errors per-schedule.

#### `executeSchedule()` (lines 107-176)

On **success** (lines 108-138):
- Runs in a DB transaction.
- Creates an agent task via `createTaskExtended()` with: `taskTemplate` as the task description, `source: "schedule"`, tags `["scheduled", "schedule:<name>"]`, and passes through `taskType`, `priority`, `targetAgentId`, `model`, `scheduleId`, `createdByAgentId`.
- Calculates next run via `calculateNextRun()`.
- Updates the schedule: sets `lastRunAt`, `nextRunAt`, resets `consecutiveErrors` to 0, clears `lastErrorAt` and `lastErrorMessage`.

On **failure** (lines 139-175):
- Increments `consecutiveErrors`.
- Truncates error message to 500 chars (`line 159`).
- If errors reach `MAX_CONSECUTIVE_ERRORS` (5), auto-disables the schedule (`enabled = false`) at line 163-167.
- Otherwise, applies exponential backoff to `nextRunAt` using the backoff table (lines 88-94): 1min, 5min, 15min, 30min, 1hr cap.

#### `calculateNextRun()` (lines 66-85)

- If `cronExpression` exists: parses with `cron-parser`'s `CronExpressionParser.parse()`, using the schedule's `timezone`, and returns `.next().toISOString()`.
- If `intervalMs` exists: adds interval to `fromTime` and returns ISO string.
- Throws if neither is set.

#### `recoverMissedSchedules()` (lines 19-58)

Called once at startup. For each due schedule:
- Skips if `nextRunAt` is absent.
- Skips if missed by less than 15 seconds (timing jitter threshold, line 26).
- Creates ONE catch-up task with an extra `"recovered"` tag (line 38).
- Calculates and sets the next run time.
- Runs the create + update in a DB transaction.

#### `runScheduleNow()` (lines 237-272)

Manual trigger endpoint:
- Looks up the schedule by ID; throws if not found or disabled.
- Creates a task with an extra `"manual-run"` tag (line 254).
- Updates `lastRunAt` only -- does NOT update `nextRunAt`, so the regular schedule continues unaffected (line 262-263).
- Runs in a transaction.

### Data Flow: Schedule Execution Cycle

1. **Creation**: MCP `create-schedule` tool validates input, calculates initial `nextRunAt`, calls `createScheduledTask()` in `db.ts` which inserts into `scheduled_tasks`.
2. **Polling**: `startScheduler()` in `scheduler.ts` sets a 10s interval calling `processSchedules()`.
3. **Due detection**: `getDueScheduledTasks()` queries `WHERE enabled = 1 AND nextRunAt <= now`.
4. **Execution**: `executeSchedule()` creates an agent task via `createTaskExtended()` with `source: "schedule"` and `scheduleId` linking.
5. **Advancement**: After task creation, `calculateNextRun()` computes the next occurrence and `updateScheduledTask()` persists it.
6. **Error handling**: Failures increment `consecutiveErrors`, apply backoff delays, and auto-disable after 5 consecutive errors.
7. **Recovery**: On startup, `recoverMissedSchedules()` fires one catch-up task per overdue schedule (missed by >15s).

### Key Patterns

- **Polling loop with mutex**: `isProcessing` flag at `scheduler.ts:12` prevents overlapping poll cycles.
- **Transactional atomicity**: Task creation and schedule updates are wrapped in `getDb().transaction()` to ensure both succeed or both roll back (`scheduler.ts:109`, `scheduler.ts:247`).
- **Exponential backoff**: Error backoff table at `scheduler.ts:88-94` with 5 tiers from 1min to 1hr cap.
- **Circuit breaker**: Auto-disable after `MAX_CONSECUTIVE_ERRORS` (5) at `scheduler.ts:96,163`.
- **Boolean/integer bridging**: `enabled` stored as SQLite integer 0/1, converted to boolean in `rowToScheduledTask()` and back in create/update functions.
- **JSON-in-TEXT**: `tags` stored as JSON string in SQLite, parsed/serialized in the mapper and write functions.

### Configuration

- Scheduler polling interval: 10,000ms default, configurable via `startScheduler(intervalMs)` parameter (`scheduler.ts:182`).
- Missed schedule jitter threshold: 15,000ms (`scheduler.ts:26`).
- Max consecutive errors before auto-disable: 5 (`scheduler.ts:96`).
- Backoff tiers: 1m, 5m, 15m, 30m, 1h (`scheduler.ts:88-94`).
