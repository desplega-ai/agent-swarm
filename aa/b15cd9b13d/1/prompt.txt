## Linear Integration Foundation - Complete Exploration Report

### Overview
The `linear-integration-foundation` branch implements **Phase 1, Step 1** of the Linear integration for agent-swarm. This is a foundation layer providing OAuth 2.0 authentication and database infrastructure for future Linear task sync features. The integration is **fully optional** and gracefully disables when `LINEAR_CLIENT_ID` is not set.

---

### Core Components

#### 1. **Database Schema** (`src/be/migrations/008_linear_integration.sql`)
Creates three new tables plus modifies the agent_tasks table:

**`linear_oauth_tokens`** - Single-row workspace-level OAuth token storage
- Stores `accessToken`, `refreshToken`, `expiresAt`, `scope`
- Handles token lifecycle management (creation, updates)

**`linear_sync`** - Bidirectional entity mapping between Swarm and Linear
- Maps both tasks and epics (`entityType`: 'task' or 'epic')
- Tracks Linear identifiers (e.g., "SWARM-123"), URLs, sync timestamp
- `syncDirection` supports three modes: 'outbound', 'inbound', 'bidirectional' (Phase 2 readiness)
- Enforces UNIQUE constraints on (entityType, swarmId) and (entityType, linearId)

**`linear_agent_mapping`** - Agent to Linear user ID mapping
- Links swarm agents to Linear users for attribution and permission mapping
- 1:1 relationship enforced with UNIQUE constraints

**Modified `agent_tasks`** - Adds 'linear' to source CHECK constraint
- Tasks can now originate from Linear (`source IN ('mcp', 'slack', 'api', 'github', 'gitlab', 'agentmail', 'system', 'schedule', 'workflow', 'linear')`)
- Table recreation required to apply constraint change
- Includes all existing columns plus `dir` (from prior migration)

**Indexes for performance:**
- `idx_linear_sync_swarmId`, `idx_linear_sync_linearId` (compound entityType lookups)
- `idx_linear_agent_mapping_agentId` (agent lookups)

---

#### 2. **OAuth 2.0 Implementation** (`src/linear/oauth.ts`)

**PKCE Flow** (OAuth 2.0 with Proof Key for Code Exchange)
- Generates 64-character code verifier and computes SHA256 code challenge
- Uses base64url encoding per spec
- State parameter validation (10-minute TTL with automatic cleanup)

**Authorization URL generation** (`getAuthorizationUrl()`)
- Redirects to `https://linear.app/oauth/authorize` with proper parameters
- Scopes: `"read,write,issues:create,comments:create,admin,app:assignable,app:mentionable"`
- Requires env vars: `LINEAR_CLIENT_ID`, `LINEAR_REDIRECT_URI`

**Token Exchange** (`handleOAuthCallback()`)
- Exchanges auth code for tokens via `https://api.linear.app/oauth/token`
- Validates state parameter against pending OAuth state map
- Stores tokens atomically in database (wrapped in transaction)
- Requires: `LINEAR_CLIENT_ID`, `LINEAR_CLIENT_SECRET`, `LINEAR_REDIRECT_URI`

**Token Refresh** (`refreshAccessToken()`)
- Uses refresh_token grant type to get new access tokens
- Automatic token refresh when expiring soon (5-minute buffer by default)
- Gracefully returns null if credentials missing

**Utility Functions**
- `getStoredTokens()` - Retrieves latest token from DB
- `isTokenExpiringSoon(expiresAt, bufferMs)` - Checks token expiration with buffer

---

#### 3. **Linear Client Wrapper** (`src/linear/client.ts`)

**Auto-refresh Caching**
- Caches LinearClient instance and access token for reuse
- Automatically refreshes token before expiry
- Resets cache on token changes

**`getLinearClient()`**
- Returns a LinearClient from @linear/sdk
- Handles token refresh transparently
- Returns null if no stored tokens (OAuth incomplete)

**`withLinearClient<T>(fn)`** - Wrapper with auto-retry
- Executes Linear API calls with automatic error handling
- On auth errors (401/403), refreshes token and retries once
- Useful for handling token expiration mid-request

**`resetLinearClient()`** - Cache invalidation
- Clears cached client state
- Forces fresh client on next call

---

#### 4. **Integration Init & Config** (`src/linear/app.ts`)

**`isLinearEnabled()`**
- Checks if `LINEAR_CLIENT_ID` is set AND `LINEAR_DISABLE` is not "true"
- Also respects `LINEAR_ENABLED` feature flag (defaults to enabled if `LINEAR_CLIENT_ID` present)

**`initLinear()`** - Idempotent initialization
- Logs warning if `LINEAR_CLIENT_SECRET` or `LINEAR_REDIRECT_URI` missing
- Returns false if integration disabled
- Safe to call multiple times

**`resetLinear()`** - State cleanup for config reloads
- Clears initialization flag
- Calls `resetLinearClient()`

---

#### 5. **HTTP Endpoints** (`src/http/linear.ts`)

Two OAuth endpoints with route definitions for OpenAPI registration:

**`GET /api/linear/authorize`**
- Redirects user to Linear OAuth consent screen
- Returns 302 redirect or error responses
- Requires: Linear integration enabled
- No API key auth required (exempt in core.ts)

**`GET /api/linear/callback`**
- Handles OAuth callback from Linear
- Parses code and state query parameters
- Exchanges code for tokens via `handleOAuthCallback()`
- Returns HTML confirmation page on success
- Returns 400 for missing/invalid params, 500 for exchange errors

**Route Definition Pattern**
- Uses `route()` function for OpenAPI spec registration
- Excludes from general API key auth bypass (security fix in PR #161 feedback)

---

#### 6. **Server Integration** (`src/http/index.ts`, `src/http/core.ts`)

**Route Handler Chain**
- `handleLinear()` called in main HTTP server handler chain
- Positioned after ecosystem but before webhooks

**Initialization**
- `initLinear()` called during server startup
- `resetLinear()` called on config reload (e.g., env var changes)
- Integration added to startup logs if enabled

**Auth Bypass**
- OAuth endpoints (`/api/linear/authorize`, `/api/linear/callback`) bypass API key auth
- Narrowed scope (not all `/api/linear/*`) per security feedback

### Key Design Decisions

1. **Workspace-level OAuth** - Single shared token per workspace (not per-agent)
2. **PKCE Flow** - More secure than basic auth code flow
3. **Auto-token Refresh** - Transparent to API callers via wrapper
4. **Optional Integration** - Gracefully disabled when LINEAR_CLIENT_ID unset
5. **Bidirectional Mapping** - `syncDirection` field prepared for future bi-sync features
6. **Atomic Token Storage** - Transaction-wrapped upsert prevents corruption
7. **Cache with Expiry Check** - Reduces API calls while ensuring fresh tokens

## Summary

Linear's integration model is substantially different from both GitHub Apps and Slack Apps. The most important findings for the agent-swarm integration are:

1. **Yes, a Linear app CAN be @mentioned in issues/comments** -- this requires the `app:mentionable` OAuth scope and the `actor=app` authorization mode.

## 3. What `app:assignable` and `app:mentionable` Mean

### `app:assignable`

When this scope is granted, the app appears in Linear's **assignee picker**. However, assigning an issue to the app does **not** make the app the assignee in the traditional sense. Instead:

- The **human user** remains the primary assignee (owner)
- The **app becomes the delegate** -- it acts on behalf of the user
- This triggers a **delegation** event, not a standard assignment

This is a deliberate design choice by Linear: "humans maintain ownership while agents act on their behalf."

### `app:mentionable`

When this scope is granted, the app appears in the **@mention autocomplete**. Users can reference the app in comments and documents, which triggers an `AgentSessionEvent` webhook.

Both scopes are **opt-in**. Existing apps without these scopes remain unchanged.

## 4. Linear Webhooks

### Two Ways to Create Webhooks

1. **Workspace-level webhooks**: Created in Settings > Administration > API > Webhooks. Requires admin privileges. Team-scoped or workspace-wide.

2. **OAuth app webhooks**: Configured in the OAuth app settings. When a workspace authorizes the app, Linear **automatically creates a webhook** for that workspace pointing to the configured URL. This is the correct approach for a distributed integration.

### Webhook HTTP Headers

Every webhook request includes:

| Header | Value |
|---|---|
| `Content-Type` | `application/json; charset=utf-8` |
| `Accept-Charset` | `utf-8` |
| `User-Agent` | `Linear-Webhook` |
| `Linear-Delivery` | Unique delivery UUID (e.g., `234d1a4e-b617-4388-90fe-adc3633d6b72`) |
| `Linear-Event` | Event type (e.g., `Issue`, `Comment`, `AgentSessionEvent`) |
| `Linear-Signature` | Hex-encoded HMAC-SHA256 signature |

### Payload Structure

```json
{
  "action": "create",
  "type": "Issue",
  "createdAt": "2025-01-24T14:32:18.084Z",
  "organizationId": "...",
  "webhookTimestamp": 1706107938084,
  "webhookId": "...",
  "url": "https://linear.app/company/issue/ENG-123",
  "data": { /* full entity object */ },
  "actor": { /* user who triggered the change */ }
}
```

For update events, the payload also includes previous values for all changed properties.

### Signature Verification Mechanism

**Algorithm**: HMAC-SHA256
**Encoding**: Hex (lowercase), **no prefix** (unlike GitHub's `sha256=` prefix)
**Header**: `Linear-Signature`
**Secret**: The webhook signing secret from the OAuth app / webhook detail page

**Verification code (Node.js / Bun)**:

```typescript
import { createHmac, timingSafeEqual } from "node:crypto";

function verifyLinearWebhook(
  rawBody: string,          // Must be raw body, NOT re-stringified JSON
  signature: string | null, // From Linear-Signature header
  signingSecret: string,
): boolean {
  if (!signature) return false;

  const computed = createHmac("sha256", signingSecret)
    .update(rawBody)
    .digest("hex");

  // Timing-safe comparison
  if (computed.length !== signature.length) return false;
  return timingSafeEqual(
    Buffer.from(computed),
    Buffer.from(signature),
  );
}
```

**Replay protection**: The JSON body contains `webhookTimestamp` (UNIX milliseconds). Verify it is within ~60 seconds of your system time.

**Critical detail**: You MUST use the **raw request body** for signature computation. Re-stringifying parsed JSON will likely produce different output and fail verification.

### AgentSessionEvent Webhooks

When the "Agent session events" category is enabled, the app receives `AgentSessionEvent` webhooks for:
- **Delegation**: User assigns/delegates an issue to the app
- **Mention**: User @mentions the app in a comment
- **Prompt**: User provides additional input to an active agent session

The payload uses `payload.agentSession` (not `payload.data`), and the `Linear-Event` header is `AgentSessionEvent`.

**Responsiveness requirement**: When you receive a `created` event, you must send an activity update or set an `externalUrls` within **10 seconds**, or the session is marked as unresponsive.

### Agent Session States

Sessions have 5 states: `pending`, `active`, `error`, `awaitingInput`, `complete`.

## 6. How Linear Setup Differs from GitHub App and Slack App

### Credential Model Comparison

| Aspect | GitHub App | Slack App | Linear App |
|---|---|---|---|
| **Identity credentials** | App ID + Private Key (PEM) | Bot Token + App Token | Client ID + Client Secret |
| **Webhook verification** | `x-hub-signature-256` header, `sha256=` prefixed HMAC-SHA256 | `x-slack-signature` header, `v0=` HMAC-SHA256 with versioned payload | `Linear-Signature` header, plain hex HMAC-SHA256 |
| **Webhook secret env var** | `GITHUB_WEBHOOK_SECRET` | `SLACK_SIGNING_SECRET` | `LINEAR_SIGNING_SECRET` |
| **Auth model** | JWT -> Installation Token (1hr, per-repo) | Bot token (long-lived) + App token (Socket Mode) | OAuth access token (24hr) + refresh token |
| **Bot identity** | GitHub App acts as bot user automatically | Slack bot user created on install | `actor=app` creates a dedicated bot user |
| **Connection method** | Webhooks (HTTP POST) | Socket Mode (WebSocket) OR Events API (HTTP) | Webhooks (HTTP POST) |
| **Total env vars needed** | 3-4 (`APP_ID`, `PRIVATE_KEY`, `WEBHOOK_SECRET`, `BOT_NAME`) | 2-3 (`BOT_TOKEN`, `APP_TOKEN`, `SIGNING_SECRET`) | 3-4 (`CLIENT_ID`, `CLIENT_SECRET`, `SIGNING_SECRET`, `REDIRECT_URI`) |

### Token Refresh Comparison

| Platform | Token Lifetime | Refresh Mechanism |
|---|---|---|
| GitHub | Installation tokens: 1 hour | Generate new via JWT + private key (no refresh token) |
| Slack | Bot token: effectively permanent | No refresh needed (revoke and reinstall to rotate) |
| Linear | Access token: 24 hours | Standard OAuth2 refresh token flow to `https://api.linear.app/oauth/token` |

### Key Architectural Differences

**GitHub App**: Uses asymmetric crypto (RSA private key) for authentication. The app generates JWTs signed with its private key, then exchanges them for short-lived installation tokens. Most complex credential management.

**Slack App**: Uses two pre-generated long-lived tokens. Socket Mode means no public endpoint needed for events (WebSocket connection initiated from your server). Simplest credential model.

**Linear App**: Uses standard OAuth 2.0 with refresh tokens. Closest to a "normal" web app OAuth flow. The `actor=app` mode is unique -- it creates a workspace-level bot identity, similar to GitHub's automatic bot user but achieved through the OAuth flow rather than app installation. Token refresh is the main operational concern (24hr access tokens).

### Webhook Verification Implementation Differences

**GitHub** (current codebase pattern in `src/github/app.ts`):
- Signature in `x-hub-signature-256` header
- Format: `sha256=<hex>` (must strip prefix before comparing)
- Uses Web Crypto API for HMAC

**Linear** (what needs to be built):
- Signature in `Linear-Signature` header (note: lowercase in practice)
- Format: `<hex>` (plain, no prefix -- simpler)
- Same HMAC-SHA256 algorithm
- Additional replay protection via `webhookTimestamp` in body

The Linear verification is actually simpler than GitHub's because there's no signature prefix to strip.

## 7. Can Issues Be Assigned to a Linear App/Bot Actor?

**Yes, but with an important nuance.** Linear uses a **delegation model**, not direct assignment:

- When a user "assigns" an issue to an app with `app:assignable`, the app becomes the **delegate**, not the assignee
- The original user remains the **assignee/owner**
- The delegation triggers an `AgentSessionEvent` webhook with a `created` action
- The app is expected to work on the issue and update the `AgentSession` state accordingly

This is intentionally different from how GitHub or Slack handle bot assignments. Linear's model preserves human accountability while allowing bots to do work.

### Practical Implications for agent-swarm

For the agent-swarm integration, the entry points for creating tasks are:

1. **Delegation (assignment)**: User assigns issue to the app -> `AgentSessionEvent` webhook -> create swarm task
2. **@Mention**: User mentions the app in a comment -> `AgentSessionEvent` webhook -> create swarm task or respond
3. **Generic webhooks**: Subscribe to `Issue`, `Comment`, etc. for broader sync (not agent-specific)

The cleanest approach (matching Taras's comment "assigned to app or via tag") is to use `AgentSessionEvent` webhooks as the primary trigger, which fire only when the app is explicitly invoked by a user. This avoids the complexity of syncing ALL issues and keeps the integration scoped to intentional interactions.

- `linear_oauth_tokens` — OAuth state (access_token, refresh_token, expiresAt)
- `linear_sync` — Bidirectional entity mapping (task ↔ Linear issue, epic ↔ Linear project)
- `linear_agent_mapping` — Agent ↔ Linear user association

### 3. **Integration Pattern: How GitHub/GitLab Work**

Both GitHub and GitLab follow the **same webhook → task creation → task execution** pattern:

**Webhook Flow:**
1. **Receive**: POST `/api/github/webhook` or `/api/gitlab/webhook`
2. **Verify**: HMAC signature (GitHub) or token header (GitLab)
3. **Route**: By event type (`pull_request`, `issues`, `merge_request`, `issue`, etc.)
4. **Detect**: Bot mentions/assignments using `detectMention()` or `isBotAssignee()`
5. **Create Task**: Call `createTaskExtended()` with VCS metadata
6. **Emit**: Workflow trigger events via `workflowEventBus`

**Handler Function Pattern:**
```typescript
export async function handlePullRequest(event: PullRequestEvent): Promise<{ created: boolean; taskId?: string }>
```

All handlers:
- Extract provider/repo/number/author/url from webhook payload
- Call `createTaskExtended(taskDescription, { vcsProvider, vcsRepo, vcsEventType, vcsNumber, vcsAuthor, vcsUrl })`
- Return `{ created: boolean; taskId?: string }`

**Deduplication:** Simple in-memory map with 60-second TTL to avoid duplicate task creation from retried webhooks.

| **GitHub** | JWT + Installation Token | `src/github/app.ts` — Generates app JWT, exchanges for installation token, caches with TTL |
| **GitLab** | PAT + Token Verification | `src/gitlab/auth.ts` — Simple token validation, stores API token for GraphQL calls |
| **Linear** | OAuth 2.0 + PKCE | `src/linear/oauth.ts` — Full PKCE flow, auto-refresh expired tokens |

**Initialization Pattern:**
```typescript
// All follow this pattern:
export function is{Provider}Enabled(): boolean
export function init{Provider}(): boolean
export function reset{Provider}(): void
```

All are called during HTTP server startup in `/src/http/index.ts`.

### 8. **HTTP Server Routing**

The main dispatcher (`/src/http/index.ts`) routes handlers in order:
```typescript
() => handleLinear(req, res, pathSegments),      // OAuth flows
() => handleWebhooks(req, res, pathSegments),    // GitHub, GitLab, AgentMail webhooks
```

The webhooks handler is a single function that **dispatches by provider**:
```typescript
if (githubWebhook.match(req.method, pathSegments)) { /* GitHub logic */ }
if (gitlabWebhook.match(req.method, pathSegments)) { /* GitLab logic */ }
if (agentmailWebhook.match(req.method, pathSegments)) { /* AgentMail logic */ }
```

**No common interface/base class** — each provider has its own if-statement.

### 9. **Type System**

**Types are split by provider:**
- `src/github/types.ts` — `PullRequestEvent`, `IssueEvent`, `CheckRunEvent`, etc.
- `src/gitlab/types.ts` — `MergeRequestEvent`, `IssueEvent`, `PipelineEvent`, etc.
- `src/types.ts` — Shared `AgentTask` schema with `vcsProvider: enum(["github", "gitlab"]).optional()`

**No union type** like `VcsEvent = GitHubEvent | GitLabEvent`. Each provider's handlers accept its own event type.

### 10. **Workflow Event Bus Integration**

Both GitHub and GitLab emit to a shared **event bus** for workflow automation:
```typescript
workflowEventBus.emit(`github.pull_request.opened`, { repo, number, title, ... })
workflowEventBus.emit(`gitlab.merge_request.opened`, { repo, number, title, ... })
```

This **decouples** webhook → task logic from **workflow trigger logic**. Workflows can listen to `github.*` or `gitlab.*` events without knowing how they're created.

### 11. **Proposal for Linear/Jira/Trello Abstraction**

Based on the existing pattern, here's what a **"ProjectTracker" abstraction** would look like:

**Type Extension:**
```typescript
// src/vcs/types.ts — extend to include all project trackers
export type VcsProvider = "github" | "gitlab" | "linear" | "jira" | "trello";

// OR separate into:
export type VcsProvider = "github" | "gitlab";  // version control
export type ProjectTrackerProvider = "linear" | "jira" | "trello" | "asana";
```

**Database Schema:**
```sql
-- Extend agent_tasks vcsProvider to include project trackers
CHECK(source IN ('...', 'linear', 'jira', 'trello', 'asana'))
CHECK(vcsProvider IN ('github', 'gitlab', 'linear', 'jira', 'trello'))

-- Add mapping tables per provider (Linear template):
CREATE TABLE IF NOT EXISTS jira_oauth_tokens (...)
CREATE TABLE IF NOT EXISTS jira_sync (...)
CREATE TABLE IF NOT EXISTS jira_agent_mapping (...)
```

**Handler Pattern:**
```typescript
// src/linear/webhook.ts (or src/jira/webhook.ts, etc.)
export async function handleLinearIssue(event: LinearIssueEvent): Promise<{ created: boolean; taskId?: string }>
export async function handleLinearCycleUpdate(event: LinearCycleEvent): Promise<{ created: boolean; taskId?: string }>

// In /src/http/webhooks.ts — add new provider:
if (linearWebhook.match(req.method, pathSegments)) { /* Linear webhook logic */ }
if (jiraWebhook.match(req.method, pathSegments)) { /* Jira webhook logic */ }
```

**No breaking changes** — the abstraction is **intentionally flat** (no interface inheritance). Each provider is self-contained, and new ones are added by:
1. Creating `src/{provider}/` directory
2. Adding webhook handler to `handleWebhooks()`
3. Extending `vcsProvider` enum
4. Adding provider-specific tables (oauth, sync, mapping)
5. Initializing provider in `http/index.ts`

---

### **Key Takeaways**

1. **The abstraction is NOT a formal interface** — it's a **convention** (database schema + handler pattern)
2. **No factory pattern** — providers are explicitly checked in if-statements
3. **Task metadata is provider-agnostic** — `vcsRepo`, `vcsNumber`, `vcsAuthor`, `vcsUrl` work for all
4. **Each provider has its own auth strategy** — JWT (GitHub), token (GitLab), OAuth (Linear)
5. **Database tables are per-provider** — Linear has `linear_oauth_tokens`, `linear_sync`, not a generic `oauth_tokens` table
6. **Workflow events ARE abstracted** — `github.pr.opened` vs `gitlab.mr.opened` keep the event structure parallel
7. **Linear is 80% ready** — just needs webhook receiver + event handlers + sync logic

The system is **pragmatic** — optimized for adding one new provider at a time rather than supporting unlimited providers with minimal code.

## Recommendation: `oauth4webapi`

It is the best fit for your use case. Here is the comparison:

---

### `oauth4webapi` (by Filip Skokan / panva) -- WINNER

- **Bun**: Explicitly works on Bun, Deno, Node, Cloudflare Workers. Built on Web Crypto + Fetch APIs. Zero dependencies, ESM-only.
- **PKCE**: First-class support, including S256 code challenge method.
- **Token refresh**: Has `refreshTokenGrantRequest()` and response processing built in.
- **Token storage**: None -- you store tokens yourself. This is what you want since you need SQLite-backed pluggable storage.
- **Provider-agnostic**: Completely generic. You pass authorization endpoint, token endpoint, etc. Works with any RFC-compliant OAuth 2.0 server (Linear, Jira/Atlassian, Trello, anything).
- **Maintenance**: Actively maintained. v3.8.5 published ~20 days ago (as of March 2026). OpenID Certified.
- **Philosophy**: Low-level building blocks. You compose the flow yourself, but every step (auth URL generation, PKCE verifier/challenge, token exchange, token refresh, token revocation) is a clean function call.

**Tradeoff**: It is low-level. You will write ~50-100 lines of wrapper code to build a reusable `OAuthClient` class that handles the full authorize-exchange-store-refresh cycle. But that wrapper is the same for every provider -- you just swap endpoint URLs and scopes.

---

### `arctic` (by pilcrowonpaper) -- Good but wrong abstraction level

- **Bun**: Works fine, runtime-agnostic (built on Fetch API).
- **Linear**: Has a [dedicated Linear provider](https://arcticjs.dev/providers/linear). Also has Atlassian (covers Jira). No Trello provider though.
- **PKCE**: Supported.
- **Token refresh**: Some providers expose `refreshAccessToken()`, but not all.
- **Token storage**: None.
- **Generic client**: Has `OAuth2Client` for unsupported providers, but it "strictly follows RFCs and may not work with providers that aren't spec-compliant."
- **Problem**: Arctic's value proposition is pre-configured provider classes (endpoint URLs baked in). But you need a *generic* system where adding a new tracker means adding config, not code. The per-provider wrappers are overhead you don't need. And for providers not in Arctic's list, you fall back to the generic client anyway, which is basically `oauth4webapi` with less flexibility.

---

### `simple-oauth2` (lelylan) -- Skip it

- **Last release**: v5.1.0, approximately 2 years ago (circa 2024).
- **PKCE**: Not natively supported in the core API. Was "factored out" as a separate module back in v1.2 and unclear if it is properly maintained.
- **Bun**: Uses `wreck` (HTTP client from hapi ecosystem) internally, not Fetch. Likely works under Bun's Node compat layer but not ideal.
- **Verdict**: Stale, no PKCE built-in, wrong HTTP abstraction for Bun.

---

### `@badgateway/oauth2-client` -- Honorable mention

- Has a nice `OAuth2Fetch` wrapper that auto-refreshes tokens and injects Bearer headers.
- PKCE support, token revocation (RFC 7009).
- Unclear on Bun compatibility -- advertised for "Node and browsers."
- Interesting fetch-wrapper pattern, but `oauth4webapi` is more standards-compliant and explicitly Bun-tested.

## What the architecture would look like with `oauth4webapi`

You would build one thin wrapper (~100 lines) like:

```
GenericOAuthManager
  - constructor(providerConfig: { authEndpoint, tokenEndpoint, clientId, clientSecret, scopes })
  - buildAuthorizationURL(state, codeVerifier) -> URL
  - exchangeCode(code, codeVerifier) -> { accessToken, refreshToken, expiresAt }
  - refreshToken(refreshToken) -> { accessToken, refreshToken, expiresAt }
```

Then each tracker provider (Linear, Jira, Trello) is just a config object with different endpoint URLs and scopes. Token persistence goes into your SQLite `oauth_tokens` table. A scheduled job or on-demand check handles refresh.

No per-provider OAuth code. One generic implementation. `oauth4webapi` handles all the RFC-compliant plumbing (PKCE challenge generation, token exchange HTTP calls, response parsing/validation).
