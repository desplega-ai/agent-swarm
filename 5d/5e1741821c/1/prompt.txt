# Researching

## Working Agreement

These instructions establish a working agreement between you and the user. The key principles are:

1. **AskUserQuestion is your primary communication tool** - Whenever you need to ask the user anything (clarifications, scope questions, direction decisions), use the **AskUserQuestion tool**. 2. **Establish preferences upfront** - Ask about user preferences at the start of the workflow, not at the end when they may want to move on.

3. **Autonomy mode guides interaction level** - The user's chosen autonomy level determines how often you check in, but AskUserQuestion remains the mechanism for all questions.

| **Autopilot** | Work independently, minimize AskUserQuestion, present comprehensive results at end |

| **Verbose** | Check in frequently, validate approach at each step, confirm before proceeding |

The autonomy mode is passed by the invoking command. If not specified, default to **Critical**.

### Pattern 3: Claude Adapter -- Process Spawning and Stream Parsing

**Found in**: `/Users/taras/worktrees/agent-swarm/2026-03-11-feat/codex-support/src/providers/claude-adapter.ts`

**Process spawn** (lines 63-74): Uses `Bun.spawn()` with the `claude` CLI. The command is built in `buildCommand()` (lines 76-101):

```typescript
private buildCommand(): string[] {
  const cmd = [
    "claude",
    "--model", this.model,
    "--verbose",
    "--output-format", "stream-json",
    "--dangerously-skip-permissions",
    "--allow-dangerously-skip-permissions",
    "--permission-mode", "bypassPermissions",
    "-p", this.config.prompt,
  ];
  if (this.config.additionalArgs?.length) cmd.push(...this.config.additionalArgs);
  if (this.config.systemPrompt) cmd.push("--append-system-prompt", this.config.systemPrompt);
  return cmd;
}
```

**Stream parsing** (lines 113-202): Reads stdout line-by-line as newline-delimited JSON. Parses `stream-json` format:
- `json.type === "system" && json.subtype === "init"` yields `session_id`
- `json.type === "result"` yields cost/token data from `json.total_cost_usd`, `json.usage`, `json.duration_ms`, `json.num_turns`

**Task file pattern** (lines 17-39): Writes a JSON file to `/tmp/agent-swarm-task-{pid}.json` containing `{taskId, agentId, startedAt}`. This file is read by the Claude hook process via the `TASK_FILE` env var.

**Stale session retry** (lines 269-318): If the process fails with "session not found" and `--resume` was used, it strips `--resume` and retries with a fresh session automatically.

**Key aspects**:
- Spawns an external CLI process (`claude`)
- Parses `stream-json` stdout for events
- Hooks are external (Claude's hook system reads `TASK_FILE` env var)
- MCP tools are discovered by Claude via `.mcp.json` on disk
- Resume uses `--resume <sessionId>` CLI flag

### Pattern 4: Pi-mono Adapter -- Library-based Session

**Found in**: `/Users/taras/worktrees/agent-swarm/2026-03-11-feat/codex-support/src/providers/pi-mono-adapter.ts`

Pi-mono does NOT spawn a subprocess. It uses the `@mariozechner/pi-coding-agent` library directly (imported at line 17-23).

**Session creation** (lines 359-433 in `PiMonoAdapter.createSession()`):
1. Creates `AGENTS.md` symlink to `CLAUDE.md` (pi-mono reads `AGENTS.md`, not `CLAUDE.md`)
2. Discovers MCP tools via `McpHttpClient` -- performs an MCP handshake over HTTP and converts them to pi-mono `ToolDefinition[]`
3. Resolves model string to pi-ai Model object (maps shortnames like `opus` -> `anthropic/claude-opus-4-20250514`)
4. Creates a swarm hooks extension via `createSwarmHooksExtension()`
5. Creates a `DefaultResourceLoader` with `appendSystemPrompt` and `extensionFactories`
6. Calls `createAgentSession(sessionOptions)` from the library

**Event normalization** (lines 181-254): Subscribes to `AgentSessionEvent` and maps:
- `message_end` -> `raw_log` (with Claude-compatible JSON structure)
- `tool_execution_start` -> `raw_log` (tool_use format)
- `tool_execution_end` -> `raw_log` (tool_result format)
- `auto_retry_start` -> `raw_stderr`

**Key differences from Claude**:
- In-process library, no subprocess
- MCP tools discovered programmatically via `McpHttpClient` (no `.mcp.json` file needed at runtime)
- Hooks are extension-based (pi-mono extension API), not file-based
- Model resolution maps shortnames to `provider/model-id` pairs
- Creates `AGENTS.md` symlink so pi-mono can read the equivalent of `CLAUDE.md`
- `canResume()` checks `SessionManager` for stored sessions

## Key Findings Summary

**1. JSON-RPC Protocol**: "JSON-RPC lite" -- same request/response/notification structure as JSON-RPC 2.0 but the `"jsonrpc":"2.0"` field is omitted. Framed as JSONL over stdio. Handshake starts with `initialize` -> `initialized` before any other method.

**2. Spawning**: `codex app-server` as a long-lived child process. Default: stdio. Experimental: `--listen ws://127.0.0.1:4500`. The TypeScript SDK (`@openai/codex-sdk`) wraps this -- spawns the CLI binary and exchanges JSONL over stdin/stdout.

**3. Thread lifecycle**: `thread/start` (create), `thread/resume` (reopen), `thread/fork` (branch history), `thread/archive` / `thread/unarchive`, `thread/list`, `thread/compact/start`, `thread/rollback`.

**4. Turn management**: `turn/start` (begin generation with user input), `turn/steer` (inject input mid-turn without starting new), `turn/interrupt` (cancel in-flight turn, status becomes "interrupted").

**5. Event streaming**: Item lifecycle is always `item/started` -> zero or more `item/*/delta` -> `item/completed`. Turn lifecycle: `turn/started` -> `turn/completed`. Thread events: `thread/started`, `thread/archived`, `thread/unarchived`, `thread/closed`, `thread/status/changed`. Clients can opt out of specific notifications via `initialize.params.capabilities.optOutNotificationMethods`.

**6. MCP integration**: Configured in `config.toml` under `[mcp_servers.<name>]`. Supports stdio (subprocess) and StreamableHttp transports. Tool names prefixed `mcp__<server_name>__<tool>`. Has `enabled_tools`/`disabled_tools` allow/deny lists. Approval via `tool/requestUserInput` with Accept/Decline/Cancel options.

**7. Authentication**: Three modes -- `apiKey` (supply OPENAI_API_KEY), `chatgpt` (managed OAuth), `chatgptAuthTokens` (external tokens from host app). For Docker/headless: `printenv OPENAI_API_KEY | codex login --with-api-key`, or `codex login --device-auth`, or copy `~/.codex/auth.json`. Can force API-only with `forced_login_method = "api"`.

**8. Type generation**: `codex app-server generate-ts --out DIR` and `codex app-server generate-json-schema --out DIR`. Pass `--experimental` to include experimental methods. Output is version-pinned to the binary.

**9. Known issues**: (a) `--dangerously-bypass-approvals-and-sandbox` doesn't actually bypass sandbox for tool child processes (issue #14068), (b) 503 errors when app-server tries upstream wss://chatgpt.com endpoint while `codex exec` works (issue #14056), (c) no RPC to trigger manual compaction when auto-compact fails (issue #9142), (d) approval response RPC gap in strict protocol-only mode (issue #14192), (e) Windows crash on certain builds (issue #12962).

**10. Approval modes**: `approval_policy` in config.toml: `"untrusted"` (default, only safe read-only auto-run), `"on-request"` (model decides), `"never"` (no prompts). CLI flags: `--full-auto` (shortcut for `--sandbox workspace-write --ask-for-approval on-request`), `--dangerously-bypass-approvals-and-sandbox` (alias `--yolo`). Sandbox modes: `read-only`, `workspace-write`, `danger-full-access`.

**Item types**: `userMessage`, `agentMessage`, `plan`, `reasoning`, `commandExecution`, `fileChange`, `mcpToolCall`, `dynamicToolCall`.

**Auth RPC surface**: `account/read`, `account/login/start`, `account/logout`, `account/rateLimits/read`, plus notifications `account/updated`, `account/rateLimits/updated`, `account/chatgptAuthTokens/refresh`.

**Integration relevance for agent-swarm**: The existing `ProviderAdapter` interface (`createSession`, `canResume`, events like `session_init`, `message`, `tool_start`, `tool_end`, `result`, `error`) maps well to the app-server protocol. A Codex adapter would spawn `codex app-server`, perform the `initialize` handshake, use `thread/start` + `turn/start` to execute tasks, map item events to `ProviderEvent`, and use `turn/interrupt` for cancellation. API key auth (`OPENAI_API_KEY` env var) is the cleanest path for Docker workers.

## Summary

The `@openai/codex-sdk` (latest: **v0.112.0**, 88 dependents on npm) is a TypeScript library that wraps the bundled `codex` Rust binary and communicates via JSONL over stdin/stdout. It provides a higher-level abstraction over the App Server JSON-RPC protocol with a Thread/Turn model. However, for an adapter that needs fine-grained control (like our `ProviderSession` interface), **the raw App Server JSON-RPC approach from the existing feasibility research is the better path**.

## 1. Package Details

- **npm**: `@openai/codex-sdk` v0.112.0 (published March 11, 2026)
- **Requires**: Node.js 18+
- **Install**: `npm install @openai/codex-sdk`
- **88 dependents** on npm

## 2. How It Spawns the App Server

The SDK wraps the bundled `codex` binary from `@openai/codex`. It spawns the CLI process using Node's `child_process.spawn()` with stdio configured as `["pipe", "pipe", "inherit"]` and exchanges JSONL events over stdin/stdout. The process lifecycle is managed automatically -- you do not manually start/stop the app-server.

## 3. TypeScript API Surface

**Source files** in `sdk/typescript/src/`:
`index.ts`, `codex.ts`, `codexOptions.ts`, `thread.ts`, `threadOptions.ts`, `turnOptions.ts`, `events.ts`, `items.ts`, `exec.ts`, `outputSchemaFile.ts`

**Main class: `Codex`**
```typescript
import { Codex } from "@openai/codex-sdk";

const codex = new Codex();                    // default options
const codex = new Codex({ env: { PATH: "/usr/local/bin" } }); // custom env

// Methods:
codex.startThread(options?)    // -> Thread
codex.resumeThread(threadId)   // -> Thread (from saved session)
```

**Thread class:**
```typescript
thread.run(prompt)              // -> Turn (buffers all events, returns final result)
thread.run(inputArray)          // supports [{ type: "text", text: "..." }, { type: "local_image", path: "..." }]
thread.runStreamed(prompt)      // -> { events: AsyncGenerator<Event> } (streaming)
thread.run(prompt, { outputSchema }) // structured JSON output
```

**Turn result:**
```typescript
turn.finalResponse  // string - the agent's final text response
turn.items          // Item[] - all items produced during the turn
```

## 4. Session Management

- **Create thread**: `codex.startThread({ workingDirectory, skipGitRepoCheck })`
- **Resume thread**: `codex.resumeThread(savedThreadId)` -- threads persist in `~/.codex/sessions`
- **Continue conversation**: Call `thread.run()` repeatedly on the same Thread instance
- **Config overrides**: Use the `config` option -- SDK flattens JSON objects into dotted TOML paths and passes as `--config key=value` flags

## 5. Event Types (from `runStreamed()`)

7 event types returned by the async generator:

| Event Type | Description |
|---|---|
| `thread.started` | Thread has been created |
| `turn.started` | A turn has begun |
| `item.started` | An item (message, command, file change, etc.) began |
| `item.updated` | Incremental delta on an item |
| `item.completed` | An item finished |
| `turn.completed` | Turn finished (includes `usage` data) |
| `turn.failed` | Turn errored |
| `thread.error` | Thread-level error |

**Item types**: `agentMessage`, `reasoning`, `commandExecution`, `fileChange`, `mcpToolCall`, `webSearch`, `planUpdate`

## 6. Authentication

- The SDK inherits the Node.js process environment by default
- Set `OPENAI_API_KEY` env var, or pass custom `env` to `new Codex({ env: { ... } })`
- The SDK auto-injects its required variables (`OPENAI_BASE_URL`, `CODEX_API_KEY`) on top of provided env
- Also supports ChatGPT subscription auth (browser-based login flow -- not useful for headless/Docker)

## 7. MCP Configuration

MCP servers are **not configured via the SDK API directly**. They go through the `config.toml` file that the `codex` binary reads:

```toml
[mcp_servers.agent-swarm]
command = "..."
# OR for HTTP:
url = "http://localhost:3013/mcp"
headers = { Authorization = "Bearer ${API_KEY}" }
```

You can pass MCP config via the SDK's `config` option using dotted paths:
```typescript
const thread = codex.startThread({
  config: {
    "mcp_servers.my_server.url": "http://localhost:3013/mcp"
  }
});
```

## 10. SDK vs Raw App Server JSON-RPC -- Comparison

| Dimension | SDK (`@openai/codex-sdk`) | Raw App Server JSON-RPC |
|---|---|---|
| **Process management** | Automatic (spawns + manages codex binary) | Manual (`Bun.spawn("codex app-server")`) |
| **API style** | High-level: `thread.run(prompt)` | Low-level: `thread/start`, `turn/start`, `item/*` notifications |
| **Event granularity** | 7 event types via `runStreamed()` | Full JSON-RPC notification stream with deltas |
| **Thread control** | `startThread()`, `resumeThread()` | `thread/start`, `thread/resume`, `thread/fork`, `thread/rollback`, `thread/compact/start` |
| **Mid-turn steering** | Not exposed (Issue #12329 open) | `turn/steer` available |
| **Turn cancellation** | Not exposed (Issue #5494 open) | `turn/interrupt` available |
| **Auth** | Inherits env or custom env object | `account/login/start` JSON-RPC method |
| **MCP config** | Via config.toml / config option dotted paths | Via config.toml (same) |
| **Runtime** | Requires Node.js 18+ | Any runtime that can spawn a process and do JSONL |
| **Maturity** | "Shipped earlier, smaller surface area" per OpenAI | "First-class integration method maintained moving forward" per OpenAI |

## Verdict for Agent-Swarm Adapter

**The raw App Server JSON-RPC approach is better for our use case.** Here's why:

1. **Missing critical methods**: The SDK does not expose `turn/interrupt` (abort) or `turn/steer` (mid-task guidance) -- both mapped to our `ProviderSession` interface in the feasibility research. Issues #5494 and #12329 are open requesting these.

2. **Node.js dependency**: The SDK requires Node.js 18+. Our workers run on Bun. While Bun has high Node.js compat, it's an unnecessary compatibility risk when the App Server protocol works with any runtime.

3. **OpenAI's own guidance**: "Codex App Server will be the first-class integration method maintained moving forward. By default, clients should use Codex App Server to integrate with Codex."

4. **Less control**: The SDK abstracts away process lifecycle, which is convenient for simple use cases but means we lose control over when to restart, how to handle crashes, and how to manage long-lived server processes.

5. **Event mapping**: The SDK's 7 event types are a strict subset of what the App Server provides. The raw JSONL stream gives us `item/*/delta` for streaming, `thread/compact/start` for context preservation hooks, etc.

The SDK would be appropriate for a quick prototype or CI script, but for a production adapter that needs abort, steering, fine-grained event monitoring, and hook-equivalent behavior, the App Server JSON-RPC is the right choice.

## Known SDK Issues

- **No abort/cancel**: [Issue #5494](https://github.com/openai/codex/issues/5494) -- no way to cancel an active `thread.run()` call
- **No turn/steer**: [Issue #12329](https://github.com/openai/codex/issues/12329) -- SDK doesn't expose the app-server's `turn/steer` for injecting mid-turn messages
- **No fork/backtrack**: [Issue #4972](https://github.com/openai/codex/issues/4972) -- conversation fork/backtrack not exposed
- **Auth issues**: [Issue #7144](https://github.com/openai/codex/issues/7144) -- SDK auth not working in some configurations

## Analysis: Session Event Data Flow (ProviderEvent -> Runner -> API/DB -> UI)

### Overview

Session events flow from provider adapters (Claude CLI or pi-mono) through a normalized `ProviderEvent` union, get buffered in the runner, are batch-POSTed to the API which writes them to the `session_logs` SQLite table, and are finally polled by the dashboard UI which parses the JSON content into a chat-style message view.

### 1. ProviderEvent Type Definition (`src/providers/types.ts:18-27`)

The normalized event union that all adapters must emit:

```typescript
export type ProviderEvent =
  | { type: "session_init"; sessionId: string }
  | { type: "message"; role: "assistant" | "user"; content: string }
  | { type: "tool_start"; toolCallId: string; toolName: string; args: unknown }
  | { type: "tool_end"; toolCallId: string; toolName: string; result: unknown }
  | { type: "result"; cost: CostData; output?: string; isError: boolean; errorCategory?: string }
  | { type: "error"; message: string; category?: string }
  | { type: "raw_log"; content: string }
  | { type: "raw_stderr"; content: string }
  | { type: "custom"; name: string; data: unknown };
```

Only `raw_log` events are streamed to the DB/UI. The `content` string of a `raw_log` event is the exact line stored in the database.

### 2. Claude Adapter (`src/providers/claude-adapter.ts`)

The Claude adapter spawns `claude` CLI with `--output-format stream-json` (line 83). It reads stdout line by line and emits **every non-empty line** as a `raw_log` event (line 138):

```
this.emit({ type: "raw_log", content: trimmed });
```

The `content` field is the raw JSON line from Claude CLI's stream-json output. Claude CLI natively produces JSON objects with shapes like:

- **Init**: `{ type: "system", subtype: "init", session_id: "..." }`
- **Assistant message**: `{ type: "assistant", message: { role: "assistant", content: [...], model: "..." } }`
- **Result**: `{ type: "result", total_cost_usd: ..., usage: {...}, ... }`

The adapter also extracts `session_init` (line 209-212) and `result` (line 215-245) events by parsing the JSON, but these are in addition to the raw_log -- the raw line is always emitted.

### 3. Pi-Mono Adapter (`src/providers/pi-mono-adapter.ts`)

The pi-mono adapter subscribes to `AgentSessionEvent` from the pi-coding-agent library (line 160) and normalizes them into the **same JSON format Claude CLI produces**. This is the "magic" -- it constructs Claude-compatible JSON strings inside the `raw_log` content:

**`message_end` event** (lines 183-209): Emits assistant text messages:
```json
{
  "type": "assistant",
  "message": {
    "role": "assistant",
    "content": [{ "type": "text", "text": "..." }],
    "model": "claude-sonnet-4-20250514"
  }
}
```

**`tool_execution_start` event** (lines 212-227): Emits tool use:
```json
{
  "type": "assistant",
  "message": {
    "role": "assistant",
    "content": [{ "type": "tool_use", "id": "...", "name": "...", "input": {...} }],
    "model": "..."
  }
}
```

**`tool_execution_end` event** (lines 229-247): Emits tool results:
```json
{
  "type": "assistant",
  "message": {
    "role": "assistant",
    "content": [{ "type": "tool_result", "tool_use_id": "...", "content": "..." }]
  }
}
```

The adapter also has **deduplication**: `lastEmittedMessage` (line 147) tracks the last emitted text to skip duplicate messages across turns (line 194).

Both adapters use an **event queue** pattern: events emitted before `onEvent()` is called are buffered in `eventQueue` and flushed when a listener registers (Claude: lines 103-111/257-264; pi-mono: lines 166-179/341-348).

### 4. Runner Event Consumption (`src/commands/runner.ts:1149-1187`)

The runner registers a listener on `session.onEvent()`. Only `raw_log` events are streamed to the API:

- **`session_init`**: Saves the provider session ID to the API via `PUT /api/tasks/:id/claude-session` (line 1153)
- **`raw_log`**: Pretty-printed to console, then pushed to a **log buffer** (line 1166)
- **`raw_stderr`**: Pretty-printed to console only (not stored in DB)
- **`result`**: Cost saving is deferred to `waitForCompletion().then()` (line 1230-1241)

#### Log Buffering (`runner.ts:517-576`)

The runner uses a `LogBuffer` with two flush triggers:
- **Size**: `LOG_BUFFER_SIZE = 50` lines (line 525)
- **Time**: `LOG_FLUSH_INTERVAL_MS = 5000` ms (line 526)

On flush, `flushLogBuffer()` POSTs to `POST /api/session-logs` with:
```json
{
  "sessionId": "<runner-session-id>",
  "iteration": <number>,
  "taskId": "<task-id>",
  "cli": "claude" | "pi",
  "lines": ["<raw JSON string>", "..."]
}
```

A **final flush** occurs after `waitForCompletion()` resolves (lines 1192-1202).

### 5. API Endpoint (`src/http/session-data.ts:24-67`)

`POST /api/session-logs` validates `sessionId` (string), `iteration` (number >= 1), and `lines` (non-empty array), then calls `createSessionLogs()`.

`GET /api/tasks/:id/session-logs` returns all logs for a task via `getSessionLogsByTaskId()`.

### 6. Database Layer (`src/be/db.ts:3281-3304`)

**Table**: `session_logs` (migration `001_initial.sql:168-177`):
```sql
CREATE TABLE IF NOT EXISTS session_logs (
    id TEXT PRIMARY KEY,
    taskId TEXT,
    sessionId TEXT NOT NULL,
    iteration INTEGER NOT NULL,
    cli TEXT NOT NULL DEFAULT 'claude',
    content TEXT NOT NULL,
    lineNumber INTEGER NOT NULL,
    createdAt TEXT NOT NULL
);
```

`createSessionLogs()` inserts all lines in a **single SQLite transaction** using a prepared batch statement. Each line gets a UUID, the `content` is the raw JSON string from the `raw_log` event, and `lineNumber` is the array index within the batch (0-based).

### 7. Dashboard Consumption (`new-ui/src/components/shared/session-log-viewer.tsx`)

The UI polls `GET /api/tasks/:id/session-logs` every 5 seconds (react-query `refetchInterval: 5000` at `new-ui/src/api/hooks/use-tasks.ts:36`).

The `parseSessionLogs()` function (lines 46-125) parses each `SessionLog.content` as JSON and expects this structure:

```typescript
{
  type?: string,           // "assistant" determines role
  message?: {
    role?: string,         // "assistant" or other
    content?: ContentBlock[] | string,
    model?: string
  }
}
```

Where `ContentBlock` is one of:
- `{ type: "text", text: "..." }` -- rendered as a text paragraph
- `{ type: "thinking", thinking: "..." }` -- rendered as collapsible thinking bubble
- `{ type: "tool_use", id: "...", name: "...", input: {...} }` -- rendered as tool call bubble
- `{ type: "tool_result", tool_use_id: "...", content: "..." }` -- rendered as tool result bubble

Non-JSON lines are treated as `system` role messages with plain text content (lines 66-74).

Role determination (lines 111-112): if `parsed.type === "assistant"` OR `parsed.message.role === "assistant"`, the role is `"assistant"`, otherwise `"user"`.

## Key Findings Summary

**The core tension**: Subscription-based auth (ChatGPT Plus/Pro) gives access to fast mode and higher rate limits, but was designed for interactive browser flows. Using it in Docker requires workarounds. API key auth is officially recommended for CI/CD but loses subscription benefits (fast mode, credits, 2x rate limits).

---

### 1. ChatGPT Subscription Auth for Codex

Codex supports two auth modes: **ChatGPT OAuth** (subscription-based) and **API key**. When signed in with ChatGPT, Codex uses your subscription's included credits and rate limits. Features like **fast mode** are **only available with ChatGPT sign-in** -- API key auth uses standard API pricing instead.

- Plus ($20/mo): 30-150 local tasks/5hrs with weekly limits, Codex agent access
- Pro ($200/mo): Unlimited GPT-5.1, largest context windows, Codex agents, early access
- All ChatGPT plans get **2x Codex rate limits** vs API key
- Credits system allows purchasing additional usage after hitting included limits

Sources: [Codex Pricing](https://developers.openai.com/codex/pricing/), [Authentication](https://developers.openai.com/codex/auth/)

---

### 2. `codex login` OAuth Flow

The default browser-based OAuth flow:

1. `codex login` opens browser to ChatGPT OAuth endpoint
2. Uses **PKCE** (Proof Key for Code Exchange) for security
3. Runs a **local HTTP server on port 1455** for the callback
4. After browser sign-in, callback returns access token to CLI
5. Tokens are persisted to `~/.codex/auth.json` (or OS keyring)

**For Docker**: The localhost:1455 callback is the blocker. You cannot complete this flow inside a container without port forwarding or pre-seeded credentials.

Sources: [Authentication](https://developers.openai.com/codex/auth/), [Discussion #4650](https://github.com/openai/codex/discussions/4650)

---

### 3. auth.json Format

Location: `~/.codex/auth.json` (or `$CODEX_HOME/auth.json`). File permissions: `0o600` on Unix.

Structure (inferred from docs and source code references):

```json
{
  "auth_mode": "chatgpt",
  "tokens": {
    "access_token": "<JWT>",
    "id_token": "<JWT>",
    "refresh_token": "<opaque-token>"
  },
  "last_refresh": "<ISO-8601 timestamp>"
}
```

For API key mode:
```json
{
  "auth_mode": "apikey",
  "api_key": "<sk-...>"
}
```

**Can it be volume-mounted into Docker?** Yes -- this is the officially documented approach for headless environments:

1. Complete `codex login` on a machine with a browser
2. Copy `~/.codex/auth.json` to the container's `~/.codex/auth.json`
3. Use `docker cp` or volume mount

**Critical rule from CI/CD docs**: Seed auth.json **only if missing**. Never overwrite on every run -- you'd discard refreshed tokens that Codex wrote during the previous run.

Sources: [CI/CD Auth Guide](https://developers.openai.com/codex/auth/ci-cd-auth), [Discussion #4650](https://github.com/openai/codex/discussions/4650)

### 4. Token Refresh

- Codex **auto-refreshes tokens** during active use before they expire
- Access tokens have a **~1 hour lifespan** (based on JWT `exp` claim, defaults to 3600s)
- If `last_refresh` is **older than ~8 days**, Codex proactively refreshes the token bundle before continuing
- Refresh tokens are **single-use** (rotation model) -- after use, a new refresh token is issued
- After successful refresh, Codex writes new tokens + new `last_refresh` back to auth.json atomically
- On 401, Codex has a built-in refresh-and-retry path

**Token lifetime summary**:
- Access token: ~1 hour
- Refresh trigger: ~8 days since last refresh
- Full token bundle validity: ~28 days before forced refresh

Sources: [Authentication](https://developers.openai.com/codex/auth/), [CI/CD Auth](https://developers.openai.com/codex/auth/ci-cd-auth), [DeepWiki Auth System](https://deepwiki.com/openai/codex/2.5-authentication-system)

### 5. `chatgptAuthTokens` Mode (App Server)

The Codex App Server (used by VS Code extension and embeddable in custom products) supports two ChatGPT auth modes:

1. **`chatgpt`** (managed): Codex owns the OAuth flow, persists tokens, refreshes automatically
2. **`chatgptAuthTokens`** (external): Host app supplies `idToken` and `accessToken` directly

For `chatgptAuthTokens`:
- Host app sends tokens via `account/login/start` with `type: "chatgptAuthTokens"`
- Codex stores tokens in memory (not persisted to disk)
- When tokens expire, app-server sends `account/chatgptAuthTokens/refresh` request to the host
- Host must respond with fresh `idToken` and `accessToken` JWTs
- Codex retries the original request after successful refresh

**For agent-swarm**: This is the most promising path for managed Docker deployments. The swarm API could act as the "host app", manage a pool of ChatGPT OAuth tokens, and supply them to Codex workers via the app-server protocol.

Sources: [Codex App Server](https://developers.openai.com/codex/app-server/), [codex-rs/app-server README](https://github.com/openai/codex/blob/main/codex-rs/app-server/README.md)

### 6. Device Code Auth

`codex login --device-auth` uses OAuth Device Code flow (RFC 8628):

1. Codex requests a device code from `auth.openai.com`
2. Displays a URL (`https://auth.openai.com/codex/device`) and a one-time code
3. User opens URL on **any device with a browser**, enters the code
4. Codex polls the auth server every few seconds for up to **15 minutes**
5. Once authorized, exchanges for access + refresh tokens with PKCE
6. Stores tokens in `~/.codex/auth.json`

**Docker viability**: Partially viable.
- The code display requires interactive terminal (or log scraping)
- User must manually open URL and enter code on another device
- **Blocking issue**: Enterprise/Team workspaces may have device code auth **disabled by default** -- workspace admin must explicitly enable it ([Issue #9253](https://github.com/openai/codex/issues/9253))
- Another issue: device code auth can fail to display code on headless servers and loop back to selection menu ([Issue #12412](https://github.com/openai/codex/issues/12412))

Sources: [CLI Reference](https://developers.openai.com/codex/cli/reference), [Issue #9253](https://github.com/openai/codex/issues/9253), [Issue #12412](https://github.com/openai/codex/issues/12412)

## Recommendations for Agent-Swarm Docker Integration

### Short-term (simplest path)
1. Use **API key auth** (`CODEX_API_KEY` env var) -- simplest, no token management needed
2. Accept the trade-off: no fast mode, no subscription rate limits

### Medium-term (subscription auth in Docker)
1. **Pre-seed auth.json** per worker container via volume mount or Docker secret
2. Set `cli_auth_credentials_store = "file"` in config.toml
3. Ensure **one auth.json per worker** -- never share across concurrent workers (race condition #10332)
4. Use `codex exec` (non-interactive mode) for task execution
5. Periodically re-seed from a fresh login if tokens stop refreshing

### Long-term (ideal architecture)
1. Implement **`chatgptAuthTokens` mode via the App Server protocol**
2. Agent-swarm API acts as the "host app" managing token lifecycle
3. Central token refresh service avoids the single-use refresh token race condition
4. Each worker gets tokens injected via the app-server JSON-RPC protocol
5. This is the most robust path but requires implementing the app-server client protocol

## Analysis: Session Resume Lifecycle

### Overview

Session resume allows an agent to continue a previous Claude Code (or pi-mono) conversation when picking up a paused/interrupted task or when a child task inherits its parent's context. The provider's session ID is persisted to the database via the API, and on resume the runner fetches it back and passes it as a `--resume <sessionId>` CLI argument to the Claude process. The `ProviderSessionConfig.resumeSessionId` field exists in the type definition but is not actively used to drive behavior -- the actual resume mechanism is `additionalArgs` containing `--resume`.

### Entry Points

- **`src/commands/runner.ts:1423`** -- `runAgent()`, the main entry point for both lead and worker agents
- **`src/providers/types.ts:30-47`** -- `ProviderSessionConfig` type defining `resumeSessionId` field

#### 1. Session ID Capture (provider -> DB)

When a provider session starts, it emits a `session_init` event containing the session ID.

**Claude adapter** (`src/providers/claude-adapter.ts:209-211`): Parses `stream-json` stdout for a JSON line with `type: "system"`, `subtype: "init"`, and extracts `session_id`. Stores it in `this._sessionId` and emits `{ type: "session_init", sessionId }`.

#### 4. Resume Decision in the Runner

**Path A -- Resuming paused tasks** (`src/commands/runner.ts:1819-1832`):

1. If the task itself has `task.claudeSessionId` (line 1821), use that: `["--resume", task.claudeSessionId]`
2. Else if the task has `task.parentTaskId` (line 1826), fetch the parent's session ID via `fetchProviderSessionId()` and use that

**Path B -- New child tasks with parentTaskId** (`src/commands/runner.ts:2048-2065`):
