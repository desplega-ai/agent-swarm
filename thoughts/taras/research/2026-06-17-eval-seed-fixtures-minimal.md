# Eval seed fixtures — minimal-seed research

Two coupled problems Taras raised: (A) the seed `.sql` fixture is a full `sqlite3 .dump` (~1 MB) carrying all schema + `_migrations`, not just the rows to seed; (B) the run-details UI shows **25 "Tasks"** for a delegation-probe attempt (20 seeded audit-history rows + 5 real run tasks).

## A — Why fixtures must carry the whole schema today

**The import is PRE-BOOT, into a fresh empty DB.** `evals/src/swarm/sandbox.ts:395-413`:

```
// SQL-dump seeding (v6 §1.2): import BEFORE the API entrypoint starts, so
// the server's first boot forward-applies missing migrations onto the
// seeded DB and boot-time caches see the seeded rows.
...
const db = new Database("/app/data/agent-swarm-db.sqlite");
db.exec(readFileSync("/tmp/eval-seed.sql"));   // the full .dump
...
await startDetachedProcess({ command: "/api-entrypoint.sh", ... })  // API boots AFTER
```

Order: **(1)** `db.exec(dump)` runs the dump into a brand-new `/app/data/agent-swarm-db.sqlite` → **(2)** the API entrypoint starts → **(3)** the migration runner forward-applies only the **missing** migrations (it skips ones already in `_migrations` — confirmed: the runner reads `_migrations` into an `applied` map and skips applied ones).

Consequences that force the full dump:
- The DB is **empty** when the dump is imported, so the dump **must** `CREATE TABLE …` — an INSERT-only seed would fail (`no such table: agent_tasks`).
- The dump **must** include a populated `_migrations` so step (3) doesn't try to re-run migrations that re-create the already-imported tables (→ conflict/error). That's exactly why `evals/scenarios/fixtures/README.md` says "MUST include `_migrations`."
- It's seeded pre-boot specifically so the server's **boot-time caches** see the rows (seeding after boot would miss them).

**Cost of the status quo:** every fixture duplicates the entire schema, and that schema **drifts** — when a real migration changes a table, the committed `.dump` silently goes stale (its CREATE TABLE no longer matches what the live migrations produce). Fixtures are generated (`generate-*.ts`), so drift is "re-generate to fix," but it's a latent footgun and ~1 MB of noise per fixture.

## A — Fix options (ranked)

1. **Structured / INSERT-only seed, schema built from the real migrations (recommended, medium effort).**
   The fixture becomes ONLY the reference rows (INSERT-only `.sql`, or typed rows). The import command builds the schema by running the **actual migration files** (already in the API image) against the empty DB, then applies the inserts — still pre-boot, so caches see the rows. The post-boot runner then finds `_migrations` fully populated and applies nothing.
   - Pro: zero schema drift (schema is always the real migrations), tiny fixtures, the seed expresses intent (just the rows).
   - Con: the import command must run the migration runner programmatically in the sandbox before the inserts (or shell out to a "migrate" subcommand). Need a migrate-then-seed entrypoint path.

2. **Two-phase boot: migrate-only → seed → full boot (clean but more invasive).**
   Start the API in a "migrate only, don't load caches / don't serve" mode (creates schema + `_migrations`), apply the INSERT-only seed, then start the full server (caches load and see the rows). Removes drift, keeps seeds tiny, but requires an entrypoint flag the API respects.

3. **Schema-subset + data (low effort interim).**
   Keep the pre-boot import, but the fixture = only the tables the seed touches (`agent_tasks` + `_migrations` + any FK deps) + the seed INSERTs, generated by running real migrations then dumping just those tables. Much smaller than the full dump and less drift surface, but still carries (subset) schema and still drifts if those specific tables change.

**Recommendation:** Option 1 — make the seed an INSERT-only artifact and have the import build the schema from the real migrations. It eliminates drift (the original motivation behind "fixtures are generated") and shrinks fixtures to just the answer-key rows. Confirm whether the API image exposes a runnable "migrate" path (e.g. a CLI subcommand or the migration runner importable in the `bun -e` import command); if not, that's the one new affordance to add.

## B — The 25-task run-details display

**Root cause:** `evals/src/runner/index.ts:~1293` calls `client.listAllTasks()` (which `GET`s **all** `agent_tasks` from the attempt's API, including the 20 seeded reference rows) and merges into `ctxTasks` filtered only by `knownIds` (the upfront scenario task IDs, to avoid dupes). So `ctxTasks` = 20 seed history rows + the 5 run tasks (lead seed + 2 children + 2 follow-ups). That set is serialized into the `task` artifact (~:1694) and the run-details "Tasks" panel renders all 25.

The seeded rows are **reference data being audited**, not run activity — they shouldn't headline the run's task list.

**Fix options:**
- **Tag/segregate at serialization (preferred):** when building the `task` artifact, mark each task as `run` vs `seed`. Run tasks = `ctx.taskIds` ∪ tasks created during the run (identifiable by `creatorAgentId ∈ {lead, workers}`, or `source==='system'` follow-ups, or `createdAt` after the boot timestamp). Seed tasks = the pre-existing terminal history. The UI then shows "Run tasks (5)" by default with a collapsed "Seeded history (20)" section.
- **Filter in the UI:** same partition, done in the run-details Tasks component (find it under `evals/ui/`), default-hiding seed rows.
- Cheapest correctness: exclude seed tasks from the `ctxTasks` artifact entirely (keep them only for the checks that need them via `apiGet`), since the delegation rubric reads seed history through the API, not through `ctx.tasks`. But keeping them tagged is better for debugging.

Note: this is purely a display/artifact concern — scoring is unaffected (the rubric's P1–P4/Q checks filter to child/follow-up tasks by `creatorAgentId`/`parentTaskId`/`taskType`, so the seed rows don't inflate any check).

## TL;DR
- Fixtures carry the full schema because the dump is imported into an **empty DB pre-boot**; the API only migrates afterward. INSERT-only seeds need the schema to exist first → build it from the **real migrations** (Option 1) and ship seeds as just the rows. Bonus: kills schema drift.
- The 25 tasks = `listAllTasks()` pulling the 20 seeded reference rows into `ctxTasks`/the artifact. Tag run-vs-seed at serialization and segregate them in the UI. Scoring is unaffected.
