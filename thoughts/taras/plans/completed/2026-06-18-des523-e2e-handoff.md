## What the feature does (1 paragraph)

When the heartbeat detects a crashed worker, the `resume` follow-up is **pinned back to the crashed worker's own (stable-ID) agent** (status `pending`) instead of going to the role-blind unassigned pool — so it's reclaimed when that agent restarts, and no wrong-specialization worker can grab it. Genuine crash pins are tagged **`crash-recovery-pin`**. If the pin is never reclaimed within `HEARTBEAT_RESUME_PIN_GRACE_MIN` (default 10m), a heartbeat reaper (`escalateUnreclaimedResumes` in `cleanupStaleResources`, runs every sweep) atomically cancels it and creates a Lead-owned **`task.reroute.decision`** task; the Lead re-delegates via `send-task` with an explicit `agentId`. Rollback switches: `HEARTBEAT_PIN_CRASH_RESUME=0` (pin off → old pool behavior), `HEARTBEAT_RESUME_PIN_GRACE_MIN=0` (reaper off).

## What the E2E must prove (gaps the unit tests can't reach)

Unit tests already cover the logic deeply (pin Case A/B, offline→pool, re-sweep invisibility, reaper escalate-once/idempotent/budget-cap, reclaimed-not-escalated, TOCTOU guard, full tracker chain, pooled-resume-not-reaped race). The E2E adds the **live** parts:

1. A **real killed worker container** is detected (no `active_session` past threshold) → resume pinned to its stable `AGENT_ID` (not `unassigned`).
2. A **restarted** worker (same `AGENT_ID`) actually **reclaims** the `pending` pin via its real poll loop.
3. **Gone-agent:** a Lead-owned `task.reroute.decision` is created referencing the original (original NOT reassigned to Lead, resume NOT pooled); the Lead can re-delegate to another worker via `send-task(agentId=…)`.
4. **No role-blind grab:** worker B never claims A's pin.

## Key facts (don't re-derive)

- **Crash-pin tag:** `crash-recovery-pin` (`CRASH_RECOVERY_PIN_TAG` in `src/tasks/worker-follow-up.ts`); `getStalePinnedResumes` (`src/be/db.ts`) filters `tags LIKE '%"crash-recovery-pin"%' AND status='pending' AND createdAt < cutoff`.
- **Reclaim path:** worker poll → `getPendingTaskForAgent(agentId)` → `startTask` (`pending`→`in_progress`); no agent-status gate.
- **Reaper:** `escalateUnreclaimedResumes` (`src/heartbeat/heartbeat.ts`), called from `cleanupStaleResources` (EVERY sweep incl. preflight-bail + post-reboot). Budget guard fails resumes at `MAX_RESUME_GENERATIONS`. Atomic terminalize: `failPendingResumeIfUnclaimed(id, status, reason)` (conditional `UPDATE…WHERE status='pending' RETURNING`, null if reclaimed in the gap).
- **Thresholds/env:** `HEARTBEAT_STALL_NO_SESSION_MIN` (5m), `HEARTBEAT_RESUME_PIN_GRACE_MIN` (10m, `0`=off), `HEARTBEAT_PIN_CRASH_RESUME` (on, `0`=off), `HEARTBEAT_INTERVAL_MS` (90s), `MAX_RESUME_GENERATIONS` (3).
- Only the graceful `POST /close` marks an agent `offline`; a SIGKILLed worker is never auto-offlined (crash detection is task/session-based).
