The `runMigrations(db)` function at line 64 executes in five phases:

#### Phase 2: Load Migration Files (lines 76-109)

#### Phase 4: Bootstrap Existing Databases (lines 128-153)

## Analysis: REST Endpoint and MCP Tool Patterns (Config as Precedent)

### Overview

The codebase follows two parallel patterns for exposing functionality: (1) REST endpoints defined via a `route()` factory in `src/http/` files, and (2) MCP tools defined via a `createToolRegistrar()` helper in `src/tools/` files. Both patterns share the same underlying DB functions from `src/be/db.ts` but have distinct structural conventions. The config feature (`src/http/config.ts` + `src/tools/swarm-config/`) implements both sides and serves as the canonical template.

---

### Pattern 1: REST Endpoints via `route()` Factory

#### 1.1 Route Definition (`src/http/route-def.ts`)

The `route()` factory at line 65 accepts a `RouteDef` object and returns a `RouteHandle` with two methods:

- **`match(method, pathSegments)`** (line 77): Delegates to `matchRoute()` from `src/http/utils.ts:69`. Compares HTTP method (uppercased) and path segments against the route's `pattern`. The `exact` flag (default `true`) controls whether trailing segments are rejected.

- **`parse(req, res, pathSegments, queryParams)`** (line 87): Extracts and validates three parts:
  - **Path params** (line 90-98): Dynamic segments are identified by `null` entries in `pattern`. Param names are extracted from the OpenAPI-style `path` string (e.g., `{id}` in `/api/config/{id}`). Each `null` in the pattern is paired with the next param name, and the corresponding `pathSegments[i]` value is assigned.
  - **Query params** (line 102): Parsed via `Object.fromEntries(queryParams)` and validated against the `query` Zod schema.
  - **Body** (line 103): Read via `parseBody(req)` from `src/http/utils.ts:40` (buffers chunks, JSON.parse) and validated against the `body` Zod schema.
  - On `ZodError`, it calls `jsonError(res, ..., 400)` at line 112-117 and returns `null`.

Every call to `route()` also pushes the `RouteDef` into the global `routeRegistry` array at line 72. This registry is consumed by the OpenAPI generator.

## Analysis: Workflow Interpolation Engine

### Overview

There are two distinct interpolation systems in the codebase. **`template.ts`** (singular) is the runtime interpolation engine used by the workflow execution engine -- it resolves `{{dot.path.tokens}}` against a nested context object and tracks unresolved tokens. **`templates.ts`** (plural) is the workflow template instantiation system -- it resolves `{{flatKey}}` tokens against a flat variables map when turning a `WorkflowTemplate` into a concrete `WorkflowDefinition`. ### Core Implementation: `template.ts` (Dot-Path Engine)

#### 1. `interpolate()` (`template.ts:13-32`)

**Signature:**
```typescript
export function interpolate(template: string, ctx: Record<string, unknown>): InterpolateResult
```

**Return type** (`template.ts:8-11`):
```typescript
export interface InterpolateResult {
  result: string;
  unresolved: string[];
}
```

**How it works:**

- Uses `String.replace()` with regex `/\{\{([^}]+)\}\}/g` at line 15. This matches anything inside double-curly braces that is not a `}` character.
- For each match, the captured group (the path string) is trimmed and split on `"."` at line 16, producing an array of keys.
- Walks the context object one key at a time (lines 18-24). At each step:
  - If the current `value` is `null`, `undefined`, or not an object, the path is pushed to `unresolved` and the token is replaced with `""` (empty string) -- lines 19-21.
  - Otherwise, it descends: `value = value[key]` at line 23.
- After walking all keys, if the final `value` is `null` or `undefined`, the path is added to `unresolved` and replaced with `""` (lines 25-28).
- If the final value is an object, it is passed through `safeStringify()` (line 29). Otherwise, it is coerced via `String(value)`.

#### 2. `safeStringify()` (`template.ts:38-44`)

A private helper that wraps `JSON.stringify()` in a try/catch. On circular reference errors (or any error), returns `"[Circular]"` instead of throwing.

#### 3. Unresolved Token Tracking

Unresolved tokens are accumulated in a local `string[]` array (line 14). A token is considered unresolved in two cases:
- The dot-path traversal hits a non-object (null, undefined, or primitive) before reaching the final key (lines 19-21).
- The traversal completes but the final value is `null` or `undefined` (lines 25-27).

Notably, empty string `""`, `0`, and `false` are all considered **resolved** -- only `null`/`undefined` trigger unresolved. The tests at lines 161-177 of the test file confirm this.

#### 4. `deepInterpolate()` (`template.ts:50-74`)

**Signature:**
```typescript
export function deepInterpolate(
  value: unknown,
  ctx: Record<string, unknown>,
): { value: unknown; unresolved: string[] }
```

Uses a recursive `walk()` closure (lines 56-71) that:
- **Strings**: delegates to `interpolate()`, collects unresolved tokens into `allUnresolved` (lines 57-61).
- **Arrays**: maps each element through `walk()` (line 62).
- **Objects** (non-null): creates a new object, walks each value (lines 63-68).
- **Primitives** (number, boolean, null, undefined): passes through unchanged (line 70).

The unresolved arrays from every string leaf are concatenated into a single flat `allUnresolved` array. The function returns both the transformed value tree and the aggregated unresolved list.
