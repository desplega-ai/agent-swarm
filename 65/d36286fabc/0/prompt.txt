## Summary: Prompt Template Preview API and Rendering

Based on my thorough exploration of the codebase, here's exactly what the preview endpoint does:

### 1. **Preview API Endpoint** (`POST /api/prompt-templates/preview`)

**Location:** `/Users/taras/worktrees/agent-swarm/2026-03-20-fet/template-registry/src/http/prompt-templates.ts` (lines 244-258)

**Request body:**
```typescript
{
  eventType: string;
  body?: string;           // Optional custom body to preview
  variables?: Record<string, unknown>;  // Variables to interpolate
}
```

**Response (PreviewResponse):**
```typescript
{
  rendered: string;        // Final markdown string (header + body, interpolated)
  unresolved: string[];    // Array of {{variable}} tokens that couldn't be resolved
}
```

### 2. **How Preview Rendering Works**

The preview endpoint does this (line 249-256):

```typescript
const { eventType, body: customBody, variables } = parsed.body;
const definition = getTemplateDefinition(eventType);
const templateBody = customBody ?? definition?.defaultBody ?? "";
const header = definition?.header ?? "";
const composed = header ? `${header}\n\n${templateBody}` : templateBody;
const { result: rendered, unresolved } = interpolate(composed, variables ?? {});

json(res, { rendered, unresolved });
```

**Key behaviors:**
- Uses the provided `body` if supplied, else falls back to the event definition's `defaultBody`
- Prepends the `header` (always on-prefix from code definitions)
- Calls `interpolate()` to resolve `{{variable}}` syntax
- **Does NOT expand `{{@template[...]}}` references** — those are only expanded in the full `/render` endpoint (DB mode only)

### 3. **Template Composition Syntax**

Two different syntaxes exist in templates:

**`{{variable}}`** — Variable interpolation (handled by preview)
- Replaced with values from the `variables` object passed to preview
- Path-based: `{{path.to.value}}` supports nested objects
- If unresolvable, added to the `unresolved` array and replaced with empty string

**`{{@template[eventType]}}`** — Template composition (NOT handled by preview)
- Used in session templates like `system.session.lead` which contains:
  ```
  {{@template[system.agent.role]}}
  {{@template[system.agent.register]}}
  {{@template[system.agent.lead]}}
  {{@template[system.agent.filesystem]}}
  ... (8 total references)
  ```
- Only expanded in `/api/prompt-templates/render` (DB mode, lines 197-198)
- Uses recursive expansion with cycle detection and max depth (3)
- **Not expanded by the preview endpoint**

### 4. **Typical Template Structure**

Example: `system.session.lead` template body (from `/Users/taras/worktrees/agent-swarm/2026-03-20-fet/template-registry/src/prompts/session-templates.ts`):

```

{{@template[system.agent.register]}}

{{@template[system.agent.self_awareness]}}

{{@template[system.agent.system]}}`,
variables: [
  { name: "role", description: "The agent's role" },
  { name: "agentId", description: "The agent's unique identifier" },
]
```

### 5. **Variable Interpolation** (`src/workflows/template.ts`)

The `interpolate()` function (lines 13-32):
- Uses regex `/\{\{([^}]+)\}\}/g` to find `{{...}}` tokens
- Splits path by `.` and traverses the context object
- For missing/null values, records the path in `unresolved[]` and replaces with empty string
- For object values, calls `JSON.stringify()` (with circular reference handling)
- Returns `{ result: string, unresolved: string[] }`

### 6. **UI Response Type** (`new-ui/src/api/types.ts`, lines 583-586)

```typescript
export interface PreviewResponse {
  rendered: string;        // The final markdown to display
  unresolved: string[];    // List of unresolved token names for warnings/UI feedback
}
```

### 7. **Key Architectural Distinction**

- **`/api/prompt-templates/preview`** → Lightweight; only interpolates `{{variable}}` syntax, used for UI preview/composition builder
- **`/api/prompt-templates/render`** → Full resolution; expands both `{{variable}}` AND `{{@template[...]}}` references, used by workers for actual prompt generation (DB mode only)
