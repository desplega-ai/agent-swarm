# Implement Plan

A thin wrapper that invokes the `desplega:implementing` skill with autonomy controls.

## When Invoked

1. **Parse autonomy mode from arguments:**
   - Check for `--autonomy=autopilot|critical|verbose` flag
   - If plan file has `autonomy:` in frontmatter, use that as default
   - Otherwise, default to **Critical** (don't prompt - implementation is more straightforward)

2. **ALWAYS invoke the `desplega:implementing` skill:**
   - Pass the plan file path
   - Pass the autonomy mode determined above
   - Let the skill handle all implementation logic

3. **If no plan path provided:**
   - Respond with: "I need a plan file to proceed. Please provide the path to the plan you would like me to implement."

## Example Usage

```
/implement-plan thoughts/<username|shared>/plans/2026-01-14-my-feature.md
/implement-plan --autonomy=autopilot @plans/feature.md
/implement-plan --autonomy=verbose @current-plan.md
```

15: # HARNESS_PROVIDER Runtime Adapter Implementation Plan
16: 
17: ## Overview
18: 
19: Add a provider adapter architecture to `runAgent()` so runtime execution can be selected by `HARNESS_PROVIDER=claude|pi` (default `claude`) without breaking existing behavior. The Claude path must remain functionally identical, while a new pi-mono adapter is introduced with parity for hooks, skills, session lifecycle, cancellation, logs, costs, and parent/child continuity.
20: 
21: ## Current State Analysis
22: 
23: ### Runtime is currently Claude-coupled
24: - Claude invocation is hardcoded in both execution paths:
25:   - `runClaudeIteration()` command construction (`src/commands/runner.ts:1090`)
26:   - `spawnClaudeProcess()` command construction (`src/commands/runner.ts:1272`)
27:   - hardcoded `--output-format stream-json` (`src/commands/runner.ts:1096`)
28: - Claude stream parsing drives session and telemetry:
29:   - session init detection (`src/commands/runner.ts:1159`, `src/commands/runner.ts:1359`)
30:   - cost extraction from `result` events (`src/commands/runner.ts:1369`)
31:   - stale resume retry logic (`src/commands/runner.ts:1550`)
32: 
33: ### Lifecycle orchestration is reusable across providers
34: - Shared runner orchestration already exists in:
35:   - `runAgent()` (`src/commands/runner.ts:1656`)
36:   - completion/finalization in `checkCompletedProcesses()` + `ensureTaskFinished()` (`src/commands/runner.ts:1593`, `src/commands/runner.ts:240`)
37:   - cancellation scan in polling loop (`src/commands/runner.ts:2066`)
38: - Shared persistence APIs are backend-agnostic:
39:   - logs: `POST /api/session-logs` (`src/http/session-data.ts:24`)
40:   - costs: `POST /api/session-costs` (`src/http/session-data.ts:88`)
41:   - finish: `POST /api/tasks/:id/finish` (`src/http/tasks.ts:188`)
42: 
43: ### Session continuity is Claude-shaped but adaptable
44: - Task stores `claudeSessionId` (`src/types.ts:125`, `src/be/migrations/001_initial.sql:98`).
45: - Resume behavior uses `--resume` with own session first, then parent session (`src/commands/runner.ts:1980`, `src/commands/runner.ts:2136`).
46: - Session persistence endpoint is Claude-specific by name (`PUT /api/tasks/:id/claude-session`, `src/http/tasks.ts:93`).
47: 
48: ### Hook parity constraints to preserve
49: - Hooks rely on `TASK_FILE` + cancellation checks against `/cancelled-tasks` (`src/hooks/hook.ts:96`, `src/hooks/hook.ts:507`).
50: - Pre/post tool semantics are implemented in `src/hooks/hook.ts` and must still trigger at equivalent lifecycle moments for pi-backed runs.
51: 
52: ## Desired End State
53: 
54: 1. `HARNESS_PROVIDER` resolves provider at runtime with this policy:
55:    - `claude` when unset
56:    - `claude` with warning when unknown
57:    - `pi` only when explicitly configured
58: 2. Runner executes via provider adapter interface (`ClaudeAdapter`, `PiMonoAdapter`).
59: 3. Both adapters emit normalized runtime events consumed by a single persistence/finalization path.
60: 4. Claude behavior remains unchanged for existing deployments.
61: 5. Pi mode fails fast on missing required auth for selected provider family (no silent fallback).
62: 6. Parent/child continuity, pause/resume, cancellation, logs, and costs work with parity-level behavior.

15: # HARNESS_PROVIDER Runtime Adapter Implementation Plan
16: 
17: ## Overview
18: 
19: Add a provider adapter architecture to `runAgent()` so runtime execution can be selected by `HARNESS_PROVIDER=claude|pi` (default `claude`) without breaking existing behavior. The Claude path must remain functionally identical, while a new pi-mono adapter is introduced with parity for hooks, skills, session lifecycle, cancellation, logs, costs, and parent/child continuity.
20: 
21: ## Current State Analysis
22: 
23: ### Runtime is currently Claude-coupled
24: - Claude invocation is hardcoded in both execution paths:
25:   - `runClaudeIteration()` command construction (`src/commands/runner.ts:1090`)
26:   - `spawnClaudeProcess()` command construction (`src/commands/runner.ts:1272`)
27:   - hardcoded `--output-format stream-json` (`src/commands/runner.ts:1096`)
28: - Claude stream parsing drives session and telemetry:
29:   - session init detection (`src/commands/runner.ts:1159`, `src/commands/runner.ts:1359`)
30:   - cost extraction from `result` events (`src/commands/runner.ts:1369`)
31:   - stale resume retry logic (`src/commands/runner.ts:1550`)
32: 
33: ### Lifecycle orchestration is reusable across providers
34: - Shared runner orchestration already exists in:
35:   - `runAgent()` (`src/commands/runner.ts:1656`)
36:   - completion/finalization in `checkCompletedProcesses()` + `ensureTaskFinished()` (`src/commands/runner.ts:1593`, `src/commands/runner.ts:240`)
37:   - cancellation scan in polling loop (`src/commands/runner.ts:2066`)
38: - Shared persistence APIs are backend-agnostic:
39:   - logs: `POST /api/session-logs` (`src/http/session-data.ts:24`)
40:   - costs: `POST /api/session-costs` (`src/http/session-data.ts:88`)
41:   - finish: `POST /api/tasks/:id/finish` (`src/http/tasks.ts:188`)
42: 
43: ### Session continuity is Claude-shaped but adaptable
44: - Task stores `claudeSessionId` (`src/types.ts:125`, `src/be/migrations/001_initial.sql:98`).
45: - Resume behavior uses `--resume` with own session first, then parent session (`src/commands/runner.ts:1980`, `src/commands/runner.ts:2136`).
46: - Session persistence endpoint is Claude-specific by name (`PUT /api/tasks/:id/claude-session`, `src/http/tasks.ts:93`).
47: 
48: ### Hook parity constraints to preserve
49: - Hooks rely on `TASK_FILE` + cancellation checks against `/cancelled-tasks` (`src/hooks/hook.ts:96`, `src/hooks/hook.ts:507`).
50: - Pre/post tool semantics are implemented in `src/hooks/hook.ts` and must still trigger at equivalent lifecycle moments for pi-backed runs.
51: 
52: ## Desired End State
53: 
54: 1. `HARNESS_PROVIDER` resolves provider at runtime with this policy:
55:    - `claude` when unset
56:    - `claude` with warning when unknown
57:    - `pi` only when explicitly configured
58: 2. Runner executes via provider adapter interface (`ClaudeAdapter`, `PiMonoAdapter`).
59: 3. Both adapters emit normalized runtime events consumed by a single persistence/finalization path.
60: 4. Claude behavior remains unchanged for existing deployments.
61: 5. Pi mode fails fast on missing required auth for selected provider family (no silent fallback).
62: 6. Parent/child continuity, pause/resume, cancellation, logs, and costs work with parity-level behavior.

95: ## Implementation Approach
96: 
97: Use a compatibility-first extraction:
98: - First isolate existing Claude logic behind an adapter with no behavior changes.
99: - Then introduce normalized runtime events and shared telemetry persistence.
100: - Add pi adapter behind `HARNESS_PROVIDER=pi`.
101: - Add targeted parity tests and explicit env failure behavior.
