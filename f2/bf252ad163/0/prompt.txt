### Design Decisions

1. **Common templates via `{{@template[...]}}` references**: `DELEGATION_INSTRUCTION` and command suggestions are registered as `common.*` templates. The PR/issue event templates reference them with `{{@template[common.delegation_instruction]}}` and `{{@template[common.command_suggestions.github_pr]}}`. This makes them independently overridable via the DB.

2. **Conditional parts as pre-computed variables**: For templates with conditional logic (closed PR merged/not-merged, review states, comment PR vs issue, related task sections), the handler code computes the final string values BEFORE calling `resolveTemplate()` and passes them as variables. This keeps templates simple and avoids template-level conditionals.

3. **Comment template uses variables for suggestions**: Since comment command suggestions depend on whether the target is a PR or Issue, and the delegation instruction is a template ref, the comment template uses `{{command_suggestions}}` as a plain variable (not a template ref) to allow the handler to pick the right one. An alternative would have been separate event types, but this keeps it simpler.

4. **Helper functions kept and exported**: `getReviewStateInfo()` and `getCheckConclusionInfo()` remain in `handlers.ts` since they compute variable values needed before calling the resolver. They're now exported for potential reuse.</result>
